# КУРСОВАЯ РАБОТА
## Тема: Создание сайта для опросов

---

## ВВЕДЕНИЕ

Цифровизация образовательной, корпоративной и общественной среды привела к резкому росту спроса на инструменты быстрого сбора обратной связи. Опросы используются в учебных курсах, внутри команд разработки, в HR-процессах, в маркетинговых исследованиях и при принятии управленческих решений. При этом практическая ценность опроса определяется не только возможностью «задать вопрос», но и качеством всего жизненного цикла: от аутентификации пользователя и удобства интерфейса до надежного хранения данных, файловых вложений, статистики и устойчивой обработки ошибок.

В рамках курсовой работы была поставлена задача реализовать полноценную информационную систему опросов в командном формате. Команда состояла из трех участников, что позволило разделить зоны ответственности и построить проект по принципам промышленной разработки: backend, frontend и опциональные расширения. Такой формат важен с учебной точки зрения, поскольку максимально приближен к реальной разработке программных продуктов, где результат зависит не от одного модуля, а от согласованной работы нескольких направлений.

Данная работа содержит две основные главы. Первая глава является общей: в ней описываются цели проекта, единая архитектура, командное распределение задач, интерфейсы взаимодействия между компонентами, а также общие требования к качеству и функциональности. Вторая глава полностью посвящена моей части проекта — клиентскому приложению. В ней подробно рассматриваются реализованные функции, соответствующие критериям оценки: клиентское приложение, авторизация на клиенте, интеграция API, асинхронные запросы, обработка ошибок и работа с файлами.

Актуальность темы определяется несколькими факторами:
- высокой востребованностью web-систем для опросов и аналитики;
- необходимостью безопасной авторизации в публичных и корпоративных приложениях;
- важностью устойчивого пользовательского интерфейса при нестабильной сети и ошибках API;
- практической ценностью навыков интеграции frontend и backend в единую систему.

Цель работы: разработать и продемонстрировать клиент-серверную систему опросов, соответствующую учебным требованиям, с акцентом на полноценную реализацию клиентского приложения и его интеграцию с серверной частью.

Для достижения цели были поставлены задачи:
1. Сформировать общую архитектуру приложения и распределить командные роли.
2. Обеспечить интерфейсное и техническое согласование между frontend и backend.
3. Реализовать клиентскую часть с учетом требований к функциональности и UX.
4. Реализовать защищенное взаимодействие клиента с API (JWT, хранение токена, обновление сессии).
5. Подключить все необходимые методы API для работы с опросами, голосованием и файлами.
6. Реализовать асинхронные запросы с отображением состояний загрузки.
7. Внедрить централизованную обработку ошибок с UI-уведомлениями.
8. Обеспечить работу с файлами: загрузка, отображение, скачивание, валидация.
9. Провести проверку работоспособности и подготовить отчетный материал.

Практическая значимость работы заключается в получении работающего программного решения и формировании устойчивых навыков командной web-разработки: декомпозиция задач, интеграция модулей, работа с API, обработка исключений и итоговая системная документация.

# ГЛАВА 1. ОБЩАЯ ЧАСТЬ ПРОЕКТА (ДЛЯ ВСЕЙ КОМАНДЫ)

## 1.1 Постановка общей задачи проекта

Командный проект был направлен на создание системы онлайн-опросов, в которой пользователь может:
- зарегистрироваться и авторизоваться;
- просматривать список опросов;
- создавать, редактировать и удалять опросы (при наличии прав);
- проходить опросы, отправляя ответы;
- просматривать статистику по результатам;
- работать с файловыми вложениями (изображениями/дополнительными материалами).

Системе предъявлялись требования как к функциональности, так и к архитектуре: соблюдение REST-подхода, использование Entity Framework, Swagger-документация, JWT-авторизация, клиентская интеграция всех методов и корректная обработка ошибок.

## 1.2 Командное распределение ролей

Для достижения результата проект был декомпозирован на три блока:

1. Backend-часть (коллега 1):
- реализация серверного API;
- моделирование сущностей и доступ к данным через EF;
- аутентификация и авторизация на JWT;
- реализация CRUD-логики;
- серверные эндпоинты для работы с файлами и статистикой;
- документация API в Swagger.

2. Опциональные задания (коллега 2):
- расширения сверх базового задания;
- доработка архитектурных и нефункциональных аспектов;
- дополнительные возможности (в зависимости от выбранных задач).

3. Frontend-часть (моя зона ответственности):
- разработка пользовательского web-клиента;
- маршрутизация и экраны приложения;
- клиентская авторизация и хранение токенов;
- интеграция всех необходимых API методов;
- асинхронные операции и индикация загрузки;
- обработка ошибок и UI-уведомления;
- клиентская работа с файлами.

Такой формат обеспечил параллельную разработку при четком разделении ответственности и понятных точках интеграции.

## 1.3 Архитектура решения

Приложение построено по классической клиент-серверной модели:

- Серверный уровень: ASP.NET Core Web API.
- Клиентский уровень: Web-приложение на React.
- Хранилище данных: SQLite (через Entity Framework Core).
- Авторизация: JWT access token (и refresh-механизм на клиенте).
- Документация API: Swagger/OpenAPI.

С точки зрения потоков данных архитектура выглядит следующим образом:
1. Пользователь выполняет вход/регистрацию через клиент.
2. Сервер проверяет данные и возвращает токен.
3. Клиент сохраняет токен и добавляет его в Authorization для защищенных запросов.
4. Клиент обращается к API-методам опросов, файлов и статистики.
5. Сервер возвращает данные либо коды ошибок, которые корректно обрабатываются в UI.

Ключевым аспектом архитектуры стало разделение ответственности:
- backend отвечает за бизнес-правила, безопасность и целостность;
- frontend отвечает за UX, валидацию пользовательского ввода, реакцию на состояния сети и понятную визуализацию.

## 1.4 Технологический стек проекта

Общий стек проекта включал:
- C# / .NET (сервер);
- ASP.NET Core Web API;
- Entity Framework Core;
- SQLite;
- Swagger;
- JWT;
- JavaScript / React / Vite (клиент);
- CSS для интерфейсного оформления;
- SignalR (в части real-time обновлений статистики).

Выбор данного стека обусловлен следующими причинами:
- быстрая разработка и отладка;
- доступная документация;
- хорошая совместимость компонентов;
- достаточная функциональность для учебного проекта и демонстрации реальных практик.

## 1.5 Общее соответствие критериям задания

Проект как единая система покрывает обязательные требования дисциплины:
- наличие клиентского и серверного приложения;
- авторизация и защищенные методы;
- CRUD для доменных сущностей;
- соблюдение REST-подхода;
- подключенная документация API;
- работа с файлами;
- проверка/тестирование ключевой логики.

Отдельные критерии, относящиеся к frontend, детально раскрыты в главе 2, так как именно они являются моей персональной частью.

## 1.6 Интеграция модулей и контракты между командами

Чтобы команды работали независимо, но совместимо, были зафиксированы API-контракты:
- единые URL эндпоинтов;
- ожидаемые тела запросов/ответов;
- формат ошибок;
- правила передачи JWT в заголовке Authorization;
- форматы данных для вопросов, вариантов ответа, статистики;
- правила загрузки и выдачи файлов.

В процессе интеграции вносились уточнения, например:
- корректная обработка id опросов на клиенте;
- унификация формата вопросов для передачи и чтения;
- защита от некорректных локальных черновых идентификаторов;
- согласование поведения при 401/403/404.

Эти доработки являются нормальной частью жизненного цикла любого проекта и отражают практическую ценность командной работы.

## 1.7 Общие требования к качеству

К проекту применялись следующие требования качества:
- функциональная полнота по заданию;
- корректная работа в типовых пользовательских сценариях;
- предсказуемая обработка ошибок;
- устойчивость к частичным сбоям (временные сетевые проблемы, недоступность отдельных методов);
- понятность интерфейса и обратной связи;
- расширяемость кода.

Для соблюдения этих требований использовались:
- модульная структура кода;
- разделение API-слоя и UI-слоя на клиенте;
- централизованная обработка HTTP-ошибок;
- контроль доступности действий по роли/владению ресурсом;
- поэтапная ручная проверка основных сценариев.

## 1.8 Общие риски и способы их минимизации

В ходе проекта были выявлены типовые риски:
1. Неполное совпадение контрактов frontend/backend.
2. Непредвиденные форматы данных в ответах API.
3. Проблемы авторизации (просроченные или некорректные токены).
4. Сбойные сценарии при работе с файлами.
5. Локальные черновые данные, не предназначенные для серверной валидации.

Для минимизации рисков использовались:
- регулярная синхронизация между участниками;
- добавление fallback-обработки на клиенте;
- явные пользовательские уведомления о причинах ошибок;
- доработка валидации перед отправкой данных;
- очистка и нормализация данных на обоих слоях.

Общая архитектура обеспечивает работоспособность и масштабируемость решения, а согласование API-контрактов позволило объединить backend, frontend и опциональные модули в единый программный продукт.

# ГЛАВА 2. МОЯ ЧАСТЬ: КЛИЕНТСКОЕ ПРИЛОЖЕНИЕ И ПОЛЬЗОВАТЕЛЬСКАЯ ЛОГИКА

Моя персональная зона ответственности охватывала полный frontend-блок проекта. Ключевая цель состояла в том, чтобы пользователь мог выполнить весь прикладной сценарий работы с системой через удобный web-интерфейс: от входа в аккаунт до создания/прохождения опроса и работы с вложениями.

Перечень критериев, закрываемых моей частью:

- Асинхронные вызовы API (async/await, обработка загрузки) — 3 балла.

## 2.2 Реализация клиентского web-приложения (8 баллов)

Клиентское приложение реализовано как SPA на React. Структура интерфейса организована вокруг маршрутов, соответствующих пользовательским сценариям:
- главная страница;
- страница входа;
- страница регистрации;
- список опросов;
- просмотр деталей опроса;
- создание нового опроса;
- редактирование существующего опроса;
- статистика по опросу;
- профиль пользователя;
- fallback-страница «не найдено».

Ключевые интерфейсные требования:
- доступность основных действий из навигации;
- логическая последовательность экранов;
- минимизация числа лишних шагов;
- единообразие кнопок, уведомлений и сообщений об ошибках.

Особое внимание уделялось не только «красоте» UI, но и его стабильности: при ошибках API пользователь не теряет контекст, а получает ясное сообщение и возможность повторить действие.

## 2.3 Клиентская авторизация: хранение и обновление токена (5 баллов)

На клиенте реализован полноценный цикл авторизации:
1. Регистрация нового пользователя.
2. Вход по учетным данным.
3. Получение access token от сервера.
4. Сохранение токена в localStorage.
5. Автоматическая подстановка Bearer-токена в защищенные запросы.
6. Обновление access token при получении 401 (через refresh-механизм).
7. Очистка токена при logout.

Технически реализован централизованный HTTP-клиент, который:
- формирует заголовки;
- обрабатывает коды ответа;
- инициирует refresh при необходимости;
- не дублирует сетевую логику по страницам.

Эта архитектура позволяет поддерживать единое поведение авторизации и упрощает сопровождение кода.

Дополнительно реализованы защитные механизмы:
- ограничение доступа к защищенным маршрутам без токена;
- автоматический выход при критических ошибках сессии;
- предотвращение отправки невалидных токенов в API.

## 2.4 Интеграция всех API-методов (7 баллов)

Интеграция API выполнена через отдельные клиентские модули (auth, polls, votes, files), что дало следующие преимущества:
- единый стиль сетевых вызовов;
- изоляция бизнес-логики от компонентов UI;
- удобство тестирования и отладки;
- минимизация дублирования кода.

Интегрированные методы включают:
- `POST /auth/register` — регистрация;
- `POST /auth/login` — вход;
- `POST /auth/refresh` — обновление токена;
- `POST /auth/logout` — выход;
- `GET /surveys` — список опросов (с пагинацией/поиском);
- `GET /surveys/{id}` — детали опроса;
- `POST /surveys` — создание;
- `PUT /surveys/{id}` — редактирование;
- `DELETE /surveys/{id}` — удаление;
- `POST /surveys/{id}/upload-image` — загрузка файла;
- `GET /surveys/{id}/image` — получение файла;
- `POST /votes/{surveyId}` — отправка ответов;
- `GET /votes/{surveyId}/stats` — получение статистики.

Интеграция покрывает весь пользовательский цикл:
авторизация → выбор опроса → заполнение → отправка → просмотр статистики.

## 2.5 Асинхронные вызовы и состояния загрузки (3 балла)

Все API-запросы выполняются асинхронно (async/await). Для каждого сценария реализованы состояния:
- `loading` — когда данные загружаются;
- `busy` — когда выполняется операция изменения данных;
- `error` — когда запрос завершился ошибкой.

Практическое значение:
- пользователь всегда понимает, что происходит;
- уменьшается вероятность повторных нажатий и дублей запросов;
- интерфейс остается отзывчивым даже при медленной сети.

Примеры реализации:
- на страницах списка и деталей опроса отображается «Загрузка...»;
- кнопки сохранения/отправки блокируются в момент запроса;
- после успешной операции интерфейс обновляется без ручной перезагрузки.

## 2.6 Обработка ошибок и пользовательские уведомления (3 балла)

Обработка ошибок построена на сочетании централизованного и контекстного подходов:

1. Централизованно в API-клиенте:
- анализ HTTP-кода;
- нормализация текстов ошибок;
- запуск refresh при 401;
- проброс исключения в UI-компонент.

2. Локально в компонентах:
- показ toast-уведомлений;
- вывод ошибки рядом с формой;
- переключение сценария в безопасный режим (например, fallback).

Обрабатываемые случаи:
- `401 Unauthorized` — попытка обновить токен, затем повтор запроса;
- `403 Forbidden` — запрет операции (например, редактирование чужого опроса);
- `404 Not Found` — ресурс не найден;
- `400 Bad Request` — ошибки валидации;
- сетевые ошибки (`TypeError`/потеря соединения).

Важный практический эффект: приложение не «падает» из-за ошибки одного запроса, пользователь сохраняет контроль над процессом.

## 2.7 Работа с файлами: загрузка, отображение, скачивание (4 балла)

Файловая подсистема клиента включает:
- выбор изображения для опроса;
- проверку расширения и размера;
- отправку файла через FormData;
- отображение изображения в карточке/деталях;

## 2.11 Технические проблемы в процессе разработки и их решение

В процессе интеграции были обнаружены и устранены реальные проблемы.

### Проблема 1: Невалидные локальные id (`local_poll_*`) в запросах к backend
Симптом: backend возвращал `400` с ошибкой валидации id.
Причина: локальный черновик опроса не должен отправляться на сервер как integer-id.
Решение:
- добавлена проверка серверного id на клиенте;
- локальные черновики обрабатываются локально;
- исключены запросы `PUT/GET/DELETE` на backend для несерверных id.

### Проблема 2: JWT malformed в логах API
Симптом: в консоли сервера появлялась ошибка `IDX14100`.
Причина: в заголовок Authorization попадал невалидный токен (например, остаток локальной сессии).
Решение:
- очистка некорректных токенов;
- повторная авторизация;
- контроль формата токена перед отправкой.

### Проблема 3: Мерцание страницы опроса
Симптом: страница циклически перезагружалась.
Причина: неверные зависимости `useEffect`, вызывавшие повторные запросы.
Решение:
- стабилизация зависимостей эффекта;
- устранение лишних триггеров перезагрузки;
- повторная проверка сценария открытия деталей опроса.

Данные исправления повысили стабильность интерфейса и качество пользовательского опыта.

- Предусловие: id опроса локальный (`local_poll_*`).
- Шаги: выбрать файл и попытаться загрузить.
- Ожидаемый результат: клиент блокирует запрос к backend и выводит понятное сообщение.

### В.6 Обработка ошибок и UX-устойчивость

**TC-ERR-01. Ошибка 404 от API**
- Шаги: обратиться к отсутствующему ресурсу.
- Ожидаемый результат: UI показывает понятное сообщение «Ресурс не найден».

**TC-ERR-02. Ошибка 500 от API**
- Шаги: смоделировать внутреннюю ошибку сервера.
- Ожидаемый результат: пользователь видит уведомление о серверной ошибке и может повторить действие.

**TC-ERR-03. Потеря сети**
- Шаги: отключить сеть и выполнить запрос.
- Ожидаемый результат: показывается уведомление о сетевой ошибке, интерфейс не ломается.

**TC-ERR-04. Многоразовое повторение ошибки**
- Шаги: несколько раз повторить неуспешный запрос.
- Ожидаемый результат: приложение остается стабильным, уведомления не приводят к зависанию UI.

**TC-ERR-05. Проверка на мерцание страницы опроса**
- Шаги: открыть детали опроса и наблюдать поведение страницы.
- Ожидаемый результат: нет бесконечных циклов загрузки, запрос выполняется однократно по сценарию.

## Приложение Е. Как формально выйти на 20–25 страниц

Если требуется строго обеспечить диапазон 20–25 страниц по методическим требованиям, рекомендуется следующий комплект:
- Основной текст (главы + введение + заключение): 14–17 страниц.
- Скриншоты в главе 2 с подписями: 5–7 страниц.
- Приложения (тест-кейсы и матрица): 2–3 страницы.

В сумме получается целевой диапазон 20–25 страниц без искусственного «раздувания» и с сохранением содержательной ценности отчета.
